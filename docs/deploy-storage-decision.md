# Deploy & Storage Decision — Where to Host, What to Persist

*Project: live World Cup 2026 dashboard. Constraint: $0 total budget.*
*Companion to [`findings.md`](findings.md) (data source decision).*

## Executive summary

Two decisions, both driven by the **$0 budget** constraint and the workload shape (read-heavy, no per-user state, spiky traffic around match windows).

1. **Hosting: Cloudflare Workers (via the OpenNext adapter).** Chosen over Vercel for unlimited free bandwidth, no commercial-use clause, and a function quota that comfortably absorbs a YouTube-driven traffic spike. Vercel is a better developer experience but the free Hobby tier's 100 GB bandwidth cap and non-commercial clause are real risks for this project. Note: Cloudflare has two ways to host Next.js — the older Pages flow (`@cloudflare/next-on-pages`, now deprecated) and the newer Workers flow (`@opennextjs/cloudflare`). We're on the latter; see "Workers vs Pages within Cloudflare" below.
2. **Storage: none.** No persistent database. Live data lives in a 30s edge cache. Historical data is a static `history.json` regenerated by a GitHub Actions cron every ~2h and served by the CDN. The data is small, append-only, and read-mostly — a database adds complexity, cost ceiling risk, and maintenance burden without enabling anything v1 needs.

The combined system runs at **$0 indefinitely** with no payment method on file anywhere, and the failure mode of a traffic spike is "slightly stale data," not "bill arrives."

## Decision 1 — Hosting

### Options considered

| Platform | Free tier | Bandwidth | Function calls | Commercial use | Next.js DX | Chosen |
|---|---|---|---|---|---|---|
| **Cloudflare Workers (OpenNext)** | ✅ | **Unlimited** | 100k requests/day | ✅ Allowed | ⚠️ Needs `@opennextjs/cloudflare` adapter | ✅ |
| Vercel Hobby | ✅ | 100 GB/mo | 100k invocations/day soft cap | ❌ Non-commercial only | ✅ First-party | ❌ |
| Netlify | ✅ | 100 GB/mo | 125k function calls/mo | ✅ Allowed | ✅ Good | ❌ |
| GitHub Pages | ✅ | "Soft" 100 GB/mo | ❌ Static only | ✅ Allowed | ❌ Can't run `/api/live` | ❌ |
| Render / Fly.io | ⚠️ | Limited / paid | Limited / paid | Varies | ✅ | ❌ |

### Why Cloudflare won

**Unlimited bandwidth is the load-bearing feature.** This is a public dashboard whose audience is a YouTube series. A single popular video can drive a 10×–100× traffic spike for 24–48 hours, then drop back to baseline. Vercel's 100 GB cap could be eaten by one popular match-day video; Netlify's 100 GB cap has the same risk. Cloudflare Pages has no bandwidth meter on the free tier — only the **Workers** function-invocation budget counts, and static asset requests don't touch it.

**No commercial-use restriction.** Vercel's Hobby plan terms explicitly forbid commercial use. A YouTube channel that's part of the Partner Program is commercial. Even if the dashboard itself isn't monetized, it's the subject of monetized videos, which is grey enough to be a risk. Cloudflare's free tier has no equivalent clause.

**Function quota is enough.** With a 30s edge cache, the ESPN proxy gets hit at most ~2 times/minute per Cloudflare region, regardless of user count. That's ~2,880 origin requests/day across all regions in the worst case — well under the 100k Workers free quota. User-facing requests served from cache don't count as function invocations.

### What Cloudflare costs us

**Next.js compatibility is not 100%.** Cloudflare Workers doesn't natively run Next.js; the `@opennextjs/cloudflare` adapter compiles the Next.js build into a Worker script + static assets. Most features work (App Router, Server Components, API routes, ISR via cache headers) but some Vercel-specific primitives don't (Vercel's `next/image` optimization service, certain Edge Runtime features). For a read-mostly dashboard this is acceptable.

**Build is slower.** Cloudflare's build runner is generally slower than Vercel's. First deploy ~2–3 min, subsequent ~1–2 min. Not a problem at this scale.

**Logs/observability are weaker.** Vercel's per-request logging UI is best-in-class. Cloudflare's is functional but coarser. Wrangler local dev partly compensates.

### Workers vs Pages within Cloudflare

Cloudflare has *two* ways to host Next.js. Important to pick the right one:

| | Pages flow (older) | Workers flow (newer) |
|---|---|---|
| Adapter | `@cloudflare/next-on-pages` | `@opennextjs/cloudflare` |
| Status | **Deprecated** (warns on install in 2026) | Cloudflare's current recommendation |
| Build | `npx @cloudflare/next-on-pages@1` | `npx opennextjs-cloudflare build` |
| Deploy | Auto on git push, no command | `npx wrangler deploy` or `npm run deploy` |
| Config | None in repo (CF dashboard only) | `wrangler.jsonc` + `open-next.config.ts` in repo |

We're on the Workers flow. The Pages flow still works but is no longer recommended, and its dependency on Vercel's `next-on-pages` build pipeline adds a layer that the OpenNext adapter avoids.

**Migration footgun observed during this project's first deploy:** if you import a Next.js repo into Cloudflare's UI without `wrangler.jsonc` in the repo, the wrangler CLI detects the missing config on deploy and auto-runs `@opennextjs/cloudflare migrate` *on Cloudflare's build machine*. The migration's generated files (`wrangler.jsonc`, `open-next.config.ts`, etc.) never make it back to your repo. Every subsequent deploy re-runs the migration unless you replicate it locally with `npx wrangler setup` and commit the output. Worth showing in the video.

### Hybrid option (rejected for v1)

A viable alternative: deploy the Next.js app on Vercel for the DX, but split the hot path `/api/live` into a standalone Cloudflare Worker to keep bandwidth/function load off Vercel. This is the best of both worlds but adds a second moving part, a second deployment pipeline, and a CORS configuration. Not worth it for v1. Reconsider for v2 if Vercel-specific features become valuable.

## Decision 2 — Storage

### Options considered

| Storage | Free tier | Fits our data? | Operational cost | Chosen |
|---|---|---|---|---|
| **No DB — edge cache + static JSON** | ✅ | ✅ | None | ✅ |
| Cloudflare KV | 100k reads/day, 1k writes/day | ✅ | Low | ❌ Overkill |
| Cloudflare D1 (SQLite) | 5 GB, 5M reads/day | ✅ | Schema + migrations | ❌ Overkill |
| Cloudflare R2 (object storage) | 10 GB, 1M Class A ops/mo | ✅ | Bucket mgmt | ❌ Overkill |
| Supabase Postgres | 500 MB, 50k MAU | ✅ | Account + schema | ❌ Overkill |
| Neon Postgres | 0.5 GB | ✅ | Account + schema | ❌ Overkill |
| PlanetScale | ❌ Removed free tier in 2024 | n/a | n/a | ❌ |

### Why "no DB" won

**The data doesn't need a database.** Three workload buckets:

1. **Live data** — current scores, in-progress match state. Lifetime: seconds to minutes. **Storage need: none.** Edge cache (30s TTL on `/api/live`) is the storage layer. If Cloudflare's cache is cold, hit ESPN again. There is no value to persisting yesterday's "live" scores once the match ends and we have final state.
2. **Static reference data** — fixtures, group assignments, team list. Lifetime: known at build time. **Storage need: none.** Pulled from `openfootball/worldcup.json` at build time and baked into the bundle.
3. **Historical data** — completed match results, accumulating standings, top-scorer counts. Lifetime: forever. Size: tiny. ~104 matches × small JSON object ≈ <500 KB total even after the whole tournament. **Storage need: a file.** A GitHub Actions cron regenerates `history.json` every ~2h and commits it to the repo. The CDN serves it. Zero database queries.

**What a database would cost us:** schema design, migrations, a free-tier account with a credit card on file (Supabase requires one even at $0 usage), connection pooling concerns in serverless, an extra failure mode, and a second secret to rotate. All for a feature set v1 doesn't need.

**What "no DB" costs us:** we cannot do per-user features (accounts, predictions, favorites, push subscriptions). v1 is explicitly read-only and anonymous, so this is fine. v2 may reopen this decision.

### The 30s edge cache as a storage layer

The pattern is unusual enough to spell out:

```
User → Cloudflare edge → /api/live (Pages Function)
                          ↓
                  Cache-Control: public, max-age=30
                          ↓
                  Edge cache (per-region)
                          ↓ (on miss)
                       ESPN API
```

The Pages Function is stateless. Every response sets `Cache-Control: public, max-age=30`. Cloudflare's CDN honors this and serves cached responses for 30s per region before re-fetching from ESPN. **The cache is the database**, with TTL-based eviction matching the freshness requirement (live scores need to be ≤30s old to feel "live").

This pattern only works because:
- The data is non-personalized (same response for all users).
- 30s staleness is acceptable for the UX (sports scores, not stock prices).
- Read volume vastly exceeds write volume (writes = ESPN updates, ~once per goal).

If any of those break, the design has to change.

### History regeneration

```
GitHub Actions cron (every ~2h)
    ↓
fetch ESPN scoreboard for past N days
    ↓
build history.json (small, deterministic)
    ↓
commit + push to repo
    ↓
Cloudflare Pages auto-deploys
    ↓
CDN serves /history.json
```

`history.json` is just a static file in the repo. Versioned by git, deployable with the rest of the site, no separate storage to manage. If the cron job dies, the file is still there (just stale). If we want to roll back a corruption, `git revert`.

## Combined architecture diagram

```
[GitHub repo]
   ├── git push → Cloudflare build (npx opennextjs-cloudflare build + wrangler deploy) → Workers script + Assets
   └── Actions cron (every ~2h) → regenerate history.json → commit → re-deploy

[User]
   ↓
[Cloudflare Workers + CDN]
   ├── /              → static HTML/JS via Workers Assets binding (free, unlimited bandwidth)
   ├── /history.json  → static asset, CDN cached
   └── /api/live      → Worker route (Next.js Route Handler)
                         ↓ (30s edge cache, per region)
                         ESPN Site API
```

Total moving parts: 3 (GitHub repo, Cloudflare Pages, GitHub Actions cron).
Total accounts with payment methods: 0.
Total databases: 0.

## Risks and limitations

**Cloudflare quota changes.** Free tier limits can change. As of 2026-05, 100k Workers requests/day is the cap; if Cloudflare cuts this, the math changes. Mitigation: keep architecture portable enough to move to a different host in a day.

**Cache stampede on first deploy / cold region.** If a region's cache is cold and a popular video drops, every user in that region triggers a fresh ESPN fetch in parallel. Mitigation: with 30s TTL and ESPN's apparent tolerance for short bursts, this self-resolves in seconds. Could add request coalescing if it becomes a real problem.

**`history.json` size growth.** Tiny for one World Cup. If the project expands to multiple tournaments / multiple sports / multi-year history, the file approach breaks down. v1 is explicitly one tournament.

**No per-user features possible.** Anonymous, read-only is a hard ceiling. v2 features that need accounts (predictions, favorites, push) will require a real backend.

**ESPN is the single point of failure.** No fallback data source wired in. Covered in [`findings.md`](findings.md) — Sofascore is a documented v2 fallback.

## Open questions / future work

- Worth adding Cloudflare KV for ETag/Last-Modified handling on ESPN responses to reduce origin hits further?
- Should `history.json` be split into yearly/monthly files once it grows past 100 KB?
- For v2 with predictions: would Cloudflare D1 + a single Pages Function be enough, or does that justify a Supabase/Neon dependency?
